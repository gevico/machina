# 背景与挑战：为什么要重写 QEMU TCG

```
  AI 芯片设计验证流程
  RTL 仿真 ──→ ISS 仿真 ──→ FPGA 验证 ──→ 流片
  (最慢)      (瓶颈↑)      (受限)       (成本最高)

  ISS 需要高效执行 Guest 指令
  传统解释执行 vs JIT 动态翻译：性能差距 10~50x
```

| 维度 | QEMU TCG 的复杂度 |
|------|-------------------|
| 历史 | 25 年（2003 年诞生） |
| 核心代码 | ~5 万行 C |
| 支持架构 | 20+ Guest × 7 Host |
| 关键技术 | 寄存器分配、指令编码、IR 优化、JIT 执行 |
| 学习曲线 | x86 指令编码需 1-2 个月理解 |

**核心问题**：能否用 AI Agent 快速重塑这个引擎？

# Agent 怎么做决策：后端先行 + Plan Mode

```
  人类经验引导 Agent 的开发路径

  ┌──────────────────────────────────────────────────────┐
  │  不是顶层设计 → 拆分模块 → AI填空                      │
  │  而是按人类经验路径走：之前团队先做后端 → Agent 也先做后端 │
  └──────────────────────────────────────────────────────┘

  Agent 收到任务后的决策过程 (Plan Mode)

  ┌───────────┐    ┌───────────┐    ┌───────────┐
  │ 1. 分析   │───→│ 2. 规划   │───→│ 3. 讨论   │
  │ x86指令   │    │ 先做哪类  │    │ 调整优先级│
  │ 格式结构  │    │ 分几步走  │    │ 微调方案  │
  └───────────┘    └───────────┘    └─────┬─────┘
                                          │ 达成一致
                                          ▼
                                   ┌───────────┐
                                   │ 4. 执行   │
                                   │ 按流程实现 │
                                   └───────────┘
```

> 「它自己分析先实现哪部分指令，x86 的格式是什么样的，分几步走。
> 你可以微调优先级，讨论出方案，然后它按流程一步步执行。」

# 性能优化实战：寄存器固定映射的故事

```
  第一轮：做了优化反而更慢了

  思路来自 Intel Houdini：客户寄存器 → 宿主寄存器直接映射
  预期：减少上下文切换，指令 1:1 翻译
  结果：dhrystone 反而变慢！

  ┌──────────────────────────────────────────┐
  │  人类提出新思路：x86 通用寄存器不够？       │
  │  → 把 SIMD 寄存器 (XMM) 拿来做 buffer     │
  │  → 实验性功能，启动参数控制                 │
  └──────────────────────────────────────────┘

  Agent 进入 Plan Mode（消耗 100K+ TOKEN）

  ┌───────────┐    ┌───────────┐    ┌───────────┐
  │ 分析可用  │───→│ 设计映射  │───→│ 输出文档  │
  │ XMM寄存器 │    │ 方案      │    │ 修改清单  │
  │ 冲突检查  │    │ 关键操作  │    │ 标注文件  │
  └───────────┘    └───────────┘    └───────────┘
```

**启示**：优化方向要人类判断，Agent 负责工程落地和实验验证

# 性能实测与诚实声明

```
  dhrystone 200K 迭代 (RISC-V → x86-64, linux-user 模式)

  QEMU    ████████████████████████████████████████  0.4~0.5s
  tcg-rs  ████████████████████████████████          0.32s
                                           ↑
                                   观察到快 20%~30%
```

| 优化类别 | 估算贡献 | 关键技术 |
|---------|---------|---------|
| 执行循环 | ~8-10% | `next_tb_hint` + `exit_target` 缓存 |
| Guest 内存访问 | ~8-10% | 无 TLB，直接 `[R14+addr]` 寻址 |
| 数据结构 + 其他 | ~8-13% | Vec 连续存储 + 零成本抽象 |

```
  ┌──────────────────────────────────────────────┐
  │  诚实声明                                     │
  │  · 目前仅实现原版 ~30% 功能                    │
  │  · 功能少 → 代码路径短 → 快是符合预期的         │
  │  · 数据价值：验证优化方向正确                   │
  └──────────────────────────────────────────────┘
```

# AI 的边界与人类核心价值

```
  "初见即巅峰"
  ┌──────────────────────────────────────────────────┐
  │  AI 代码的第一印象很好——风格高、结构清晰            │
  │  像中级甚至接近高级工程师写的                       │
  │                                                  │
  │  但深入细究就会发现内部的具体问题                    │
  │  → 第一面就是最好的一面                            │
  └──────────────────────────────────────────────────┘
```

| AI 擅长 | AI 不擅长 |
|--------|----------|
| C → Rust 对照翻译 | 性能优化方向判断 |
| 批量生成 184 条指令函数 | 上下文变长后遗忘细节 |
| 测试编写 + 边界覆盖 | 多线程竞态条件调试 |
| 设计文档 + 代码分析 | 全局架构最优解 |

```
  人类核心价值 = 定义问题的能力 + 验证结果的能力
  ┌──────────────────────────────────────────────┐
  │  · 知识水平 ≥ AI，才能有效协作                  │
  │  · AI 倒逼学习——否则会被"欺骗"                 │
  │  · 商业级质量仍需人深度参与，AI 是工具非替代      │
  └──────────────────────────────────────────────┘
```

# 专家经验 × Agent 智力

```
  The Lever Analogy

  Model intelligence = lever length
  Expert experience  = fulcrum position

            effort                     outcome
              |                           |
              v                           v
    +---------+---------------------------+---+
    |/////////|                           |   |
    +---------+---------------------------+---+
                     ^
                     |
                +---------+
                | Fulcrum |
                | (Expert)|
                +---------+

  Stronger model -> longer lever
    -> fulcrum covers LESS area (no need for detail)
    -> but fulcrum position MORE critical (direction)
```

| 阶段 | 人的角色 | Agent 自主度 |
|------|---------|-------------|
| 早期（后端） | 深度参与，定路径、选策略 | 低，按指令执行 |
| 中期（前端+IR） | 定需求、审设计文档 | 中，自主规划+自测 |
| 后期（优化+测试） | 只管方向和验收 | 高，自主迭代 |

**模型越强，你不需要知道更多，但你需要判断得更准**

# 项目数据总览

| 指标 | 数值 | 说明 |
|------|------|------|
| 开发时间 | 5 天 | 全程 AI Agent 驱动 |
