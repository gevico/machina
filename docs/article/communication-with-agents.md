# 与 AI 协作的第一课：学会说话

> 你不需要学 Prompt Engineering，你需要学会沟通。这项技能，对人和对 AI 完全通用。

---

## 一个被忽略的事实

过去一年，"Prompt Engineering" 成了技术圈最热的词之一。各种课程、教程、模板层出不穷，仿佛这是一门全新的学问，需要从零学起。

但我在用 AI Agent 完成了好几个系统级项目之后，得出了一个可能让很多人意外的结论：

**与 AI 协作最核心的能力，不是编程，不是掌握某种 prompt 模板，而是沟通——一项你从小就在练习的技能。**

更准确地说，是两个能力：

1. **清晰表达需求**——什么时间、什么场景、什么条件下，完成什么事，得到什么结果。
2. **渐进式披露**——不要一股脑把所有东西都倒出来，而是分层、分步骤地提供信息。

这两条原则，对人和对 Agent 完全通用。

---

## 第一原则：把需求说清楚

### 人与人之间

你一定遇到过这样的需求描述：

> "把这个功能做一下。"

然后你追问了半小时：什么功能？哪个页面？触发条件是什么？给谁用的？做完长什么样？——这些本该在一开始就说清楚的事情。

好的需求描述是什么样的？

> "用户登录后，在首页，当购物车中有商品时，展示一个优惠弹窗，用户能看到折扣金额并直接跳转到结算页。"

一句话，包含了场景（登录后首页）、条件（购物车有商品）、动作（展示弹窗）、结果（看到折扣并跳转）。这就是经典的 5W1H 框架——When、Where、Who、What、Why、How。

没什么新鲜的。这是项目管理和团队协作中最基础的沟通准则。

### 对 Agent 也一样

OpenAI 最近发布了一篇关于 Agent Skills 的实践文章，其中有一个很好的对比。他们给 Agent 的 Skill 写描述（description），差的和好的版本分别长这样：

差的描述：

> "Run the mandatory verification stack."

好的描述：

> "Run the mandatory verification stack **when changes affect runtime code, tests, or build/test behavior**."

区别在哪？好的描述回答了 **when**（什么时候触发）和 **what scope**（影响范围）。差的描述只说了做什么，没说什么时候做、在什么条件下做。

这和你给同事布置任务是完全一样的逻辑。

### CLAUDE.md：给 Agent 写的 Onboarding 文档

我在做 tcg-rs（QEMU TCG 的 Rust 重写）这个项目时，花了大量时间写 `CLAUDE.md`。这个文件本质上就是给 AI Agent 写的项目手册，里面包含：

- **项目是什么**：一个动态二进制翻译引擎
- **怎么构建和测试**：具体的 cargo 命令
- **架构是什么样的**：翻译流水线、crate 结构、核心数据结构
- **代码风格约束**：80 列行宽、4 空格缩进、英文注释
- **设计原则**：不向后兼容、基于 trait 的可扩展性、最小化 unsafe

你可能会说——这不就是给新入职同事写的 onboarding doc 吗？

**对，就是。** 给 Agent 写文档和给人写文档，本质上没有任何区别。你需要让对方知道：这个项目是做什么的、怎么做、有什么约束、什么能做什么不能做。

OpenAI 的 `AGENTS.md` 也是同样的思路。他们的建议是：最高优先级规则放最前面，每条规则写成 if/then 触发器——

> "Before editing runtime or API changes, call `$implementation-strategy`."

> "Run `$code-change-verification` when runtime code, tests, examples, or build/test behavior changes."

简洁、明确、可执行。这不是什么 prompt 黑魔法，这就是好的沟通。

## 第二原则：渐进式披露

### 你不会第一次见面就把所有事说完

想象一下：你入职一家新公司，第一天，leader 把所有设计文档、历史决策记录、技术债备忘录、线上事故复盘——几百页文档一股脑甩给你，然后说"看完了来找我"。

你会崩溃。

正常的做法是什么？先讲大图——我们是做什么的、核心架构长什么样、你负责哪一块。第一周搞清楚这些就够了。遇到具体问题再深入，按需提供上下文。

**对 AI 也是如此。**

很多人在使用 AI Agent 时，会犯一个直觉性的错误：把所有能想到的信息都塞进 prompt 里。10000 字的上下文、三十条规则、五个文件的完整内容……结果 AI 的表现反而下降了——因为关键信息被淹没在噪声里。

### OpenAI Skills 的三层模型

OpenAI 在他们的 Agent Skills 系统中，把渐进式披露做成了一种架构设计。一个 Skill 的信息分三层加载：

**第一层：Metadata**

只有 name 和 description。Agent 启动时加载所有 Skill 的 metadata，用来判断"我有哪些能力"。

**第二层：SKILL.md body**

只有当 Agent 决定要使用某个 Skill 时，才加载完整的指令文档。

**第三层：Scripts / References / Assets**

只有在实际执行时，才激活脚本和资源文件。

这就是渐进式披露——**先让 Agent 知道有什么，再告诉它怎么做，最后才给它工具。** 每一层都是按需加载的，避免了上下文窗口被无关信息占满。

他们的原文说得很直接：

> "Progressive-disclosure model says those fields are what get loaded at startup for all skills. The full SKILL.md body and any scripts/, references/, or assets/ are loaded only later, when the skill is actually activated."

### 在实际对话中怎么做

渐进式披露不只是一种系统设计理念，它也是你在日常与 AI 对话时应该遵循的方法。

**不要这样做**：

> "我有一个 Rust 项目，用的是 workspace 结构，有 6 个 crate，分别是 core、backend、frontend、exec、linux-user、decode。core 里面定义了 IR 的 opcode、types、temps……（后面 3000 字省略）……现在帮我在 backend 的 x86_64 emitter 里加一个 CMOV 指令的编码。"

**应该这样做**：

第一步：

> "我需要在 x86_64 后端新增 CMOV 指令的编码。先看一下 backend/src/x86_64/emitter.rs 里现有的条件指令是怎么实现的。"

让 Agent 自己去读代码、理解模式。然后第二步：

> "参考 SETcc 和 Jcc 的实现方式，按同样的模式添加 CMOVcc。条件码复用 Cond 枚举。"

Agent 已经有了上下文，这时给出具体指令就够了。如果有特殊约束，第三步再补充：

> "注意 CMOV 的操作码是 0F 4x，需要 P_EXT 前缀标志。"

三步对话，每步都建立在前一步的理解之上。这比一次性灌入 5000 字 prompt 高效得多。

## 两个原则的融合

清晰沟通和渐进式披露不是两个孤立的技巧，它们是一体两面：

- **清晰沟通**解决的是"每一次交互的质量"——每次说话都要说到点上。
- **渐进式披露**解决的是"多次交互的节奏"——什么时候该说什么。

合在一起就是：**在正确的时机，提供正确的信息，以正确的方式。**

这对人类协作成立。对 Agent 协作同样成立。

OpenAI 博客中还有一个实践值得借鉴——**narrow contracts**。每个 Skill 只做一件事，有明确的触发条件和输出格式。这和人类团队中"职责清晰、接口明确"的分工原则一模一样。当你和 Agent 协作时，如果你一次性让它"重构整个模块并顺便写测试还有更新文档"，效果大概率不如拆成三个独立的、边界清晰的任务。

## 这不是 Prompt Engineering

我想强调的是：以上所有内容，没有一条是 AI 时代才出现的新知识。

5W1H 框架——项目管理教科书里写了几十年。

渐进式披露——UI/UX 设计领域的经典原则，Jakob Nielsen 1990 年代就在推广了。

任务拆分和职责明确——软件工程和团队管理的基本功。

所谓 "Prompt Engineering"，其实就是把这些古老的沟通智慧应用到了一个新的对象上。对象变了，人变成了 AI，但沟通的本质没有变。

**你需要做的不是去学一门新技能，而是把你已有的沟通能力迁移到新场景中。**

如果你能把需求跟同事说清楚，你就能把需求跟 Agent 说清楚。如果你懂得新人入职时循序渐进地介绍项目，你就懂得跟 AI 渐进式披露上下文。

反过来说——如果你跟 AI 沟通效率低，很可能你跟人沟通时也有同样的问题。AI 只是让这个问题变得更加显性了。

---

## 写在最后

AI 时代，技术门槛在快速降低。写代码的壁垒在消失，调用 API 的壁垒在消失，甚至系统设计的壁垒也在被 Agent 拉平。

但有一项能力的价值不降反升：**把事情说清楚的能力。**

这是一项对人、对 AI 都通用的元技能。它不会随着模型升级而过时——恰恰相反，模型越强大，清晰的需求描述就越能释放它的潜力。

别再焦虑 Prompt Engineering 了。学会说话，就够了。
