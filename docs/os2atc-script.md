# OS2ATC 2026 口播稿

> 演讲主题：tcg-rs — 一个人、两个 Agent、五天：AI 驱动的系统软件工程实践
> 演讲者：刘超（泽文）
> 总时长：18 分钟 + 2 分钟 Q&A

---

## 第 1 页：封面 [0:30]

大家好，我是泽文。今天分享的主题是 tcg-rs——一个完全由 AI Agent 驱动构建的二进制动态翻译系统。

但今天我不想讲太多技术细节。我想讲的是：**在这个过程中，人和 Agent 是怎么协作的**——Agent 怎么做决策、怎么调试、怎么优化性能，以及人类在其中扮演什么角色。

---

## 第 2 页：背景与挑战 [1:30]

先说背景。我之前在做 AI 芯片的功能模拟器。ISS——指令集模拟器——是芯片设计验证的关键环节。传统解释执行太慢，JIT 动态翻译能带来 10 到 50 倍的性能提升，但 JIT 引擎本身开发成本很高。

QEMU TCG 是这个领域的标杆——25 年历史，5 万行 C，支持 20 多种架构。光是 x86 的指令编码格式，我当年就学了一两个月。

之前三个人的团队，花了半年多才搭起架构框架。那是 ChatGPT 刚出来的年代，AI 还只能对话，没法参与编程。

恰逢 Claude、Codex 这类 Agent 工具兴起，我就想试试：能不能一个人用 AI Agent 来做这件事？

---

## 第 3 页：动态二进制翻译一图流 [1:00]

快速科普一下什么是动态二进制翻译。用一句话解释：给你一本日语书，我把你想读的章节实时用中文朗读出来。两个关键词：翻译，加实时。

技术上就是在运行时，把客户机的 RISC-V 指令动态翻译成宿主机的 x86-64 指令，然后直接执行。翻译结果缓存在 Translation Block 里，一次翻译、多次执行。

技术细节不展开了，重点看我们怎么用 Agent 来做。

> 【过渡】接下来是今天的核心内容——人和 Agent 的协作模式。

---

## 第 4 页：核心主张——人是 PM，Agent 是工程团队 [1:00]

整个项目的角色分工非常清晰。

我是产品经理，定义做什么、为什么做。Claude 是架构师兼主力开发者，负责 Plan Mode 规划、核心逻辑实现和设计文档编写。Codex 做 Code Review 和测试补全。

人在这个过程中只抓两件事：**第一，设计文档有没有和我的理解对齐。第二，测试结果有没有和我的预期一致。**

用泽文自己的话说：「保证 Agent 设计的思路和我一样，验证 Agent 交付的结果能不能保证正确性和性能。把握住这两点就可以了，剩下的就是把活都交给它。」

---

## 第 5 页：Agent 怎么做决策——后端先行 + Plan Mode [1:30]

很多人第一反应是：你是不是先设计好整个框架，再让 AI 一个模块一个模块去填？

不是的。我没有一上来把所有顶层设计规划好。

我的策略是按照自己之前人工开发的经验路径走。之前团队是先做后端指令生成——因为研究 Intel Houdini 的时候，这块最熟。所以这次也一样，先让 Agent 做 x86 后端。

我就跟它说：能不能帮我参考 QEMU，给我实现一个 x86 的后端指令生成？

然后 Claude 进入 Plan Mode——当你给它一个比较复杂的需求时，它会先切换到计划模式，分析先实现哪部分指令、x86 的指令格式是什么样的、分几步走。你可以和它微调优先级，讨论出一个方案，然后它按流程一步步执行。

第一次交付的结果给了我不小的惊喜。虽然没把所有指令都生成，但已有部分正确率挺高。

---

## 第 6 页：设计文档驱动 + 上下文管理 [1:30]

后端做出来以后，发现 Agent 整体上按 QEMU 的风格做事，但会结合 Rust 语法做一些优化和调整。

整个开发采用 CLAUDE.md 驱动的迭代模式。每次让 Agent 做完一个功能，就让它把 Plan Mode 讨论的结果转化成设计文档，同步更新进来。整个项目积累了 98KB 的设计文档。

这里有一个关键细节：**上下文过长时，模型的智力会下降。** 基本上我对话三四轮就终止掉，重新开一个新上下文。

这样做会不会丢失上下文？不会。因为有设计文档在，Agent 只需要读文档就能快速恢复。而且我尽可能让它在一次上下文里把一个模块完成，不要跨两个上下文。

用芯片设计的类比来说：我们芯片上也有设计文档，对应有 RTL 版本、功能模拟器版本。所有实现全部对齐同一份文档。AI Agent 也是一样的思路。

---

## 第 7 页：Agent 自我迭代——写测试发现 Bug 自修复 [1:15]

Agent 最让我惊喜的能力之一是自我迭代。

当它自己写测试的时候，发现功能实现有问题，它自己就会去修。大部分功能的错误，都是它自己写测试的时候发现的。而且还很多。我反而静态 review 代码，没有找出来太多。

当 Agent 发现编译错误或者校验不通过，它会反过来检查代码。你可以在配置里告诉它参考语料——模拟 RISC-V 就给它 RISC-V 的指令集手册，后端翻译成 x86 就给它 x86 的手册。它会去理解你原本预期的结果是什么样，然后反推代码逻辑的正确性。

就这么一轮一轮地迭代。所以 4.5 万行代码里，50% 是功能代码，50% 全部都是测试。这就是安全网。

---

## 第 8 页：多 Agent 协作——双终端模式 [1:00]

在这个项目里不只用了 Claude，还用了 Codex。两个 Agent 怎么协作？

很简单。在一个终端启动 Claude，在另一个终端启动 Codex。一个活干完了，另一个让它接着去 review 代码就行了。

没有什么复杂的多 Agent 调度框架——就是人在中间做调度。一个写代码，一个审查。

为什么用不同厂家的模型？让它们交叉验证，互相卷。效果比单个 Agent 好很多。另一个原因比较现实——主要是 Claude 太贵了。

现在也有一些开源工具可以做多 Agent 协同，但我觉得那种都太沉重了。最简单的方式往往是最有效的。

---

## 第 9 页：性能优化实战——寄存器映射的故事 [1:30]

这是整个项目中最有意思的一个案例。

思路来自 Intel 的 Houdini：把客户机的通用寄存器直接映射到宿主机的寄存器上。这样做指令翻译的时候，不需要中间数据的上下文切换，加减乘除可能 1:1 翻译过去。

我把这个思路告诉了 Claude，让它去实现。结果第一轮做完，跑 dhrystone **反而更慢了**。

然后我提出了一个新思路：是不是 x86 的通用寄存器不够了？把 SIMD 的 XMM 寄存器拿过来用一用——大部分场景下 SIMD 寄存器不会全部用光，可以拿来做通用寄存器的映射。

Agent 进入 Plan Mode，消耗了 100 多 K 的 TOKEN，做了完整的规划——用哪些寄存器做映射、关键操作有哪些、会修改哪些文件，标得非常清楚。

这个案例最能体现分工：**优化方向的判断需要人类经验，工程落地和实验验证交给 Agent。**

---

## 第 10 页：Agent 怎么做性能分析 [1:15]

Agent 做性能分析其实有一整套方法论。

第一层是显式指标——跑 dhrystone，用 time 统计，和 QEMU 做对比。第二层是代码质量——让 Agent 写 trace log，打印 IR 翻译情况和代码生成情况。第三层是热点分析——用 perf 看热点函数，找到以后做针对性优化。

一整套流程几乎不用人参与。它自己就能找出来。

这里就体现出通用智力强不强了。Agent 的智力越强，这些手段你越不需要告诉它。它大量的语料库里面，能够嗅到什么是好代码、什么是坏代码。

> 【过渡】来看实测结果。

---

## 第 11 页：性能实测与诚实声明 [1:00]

用 dhrystone 做基准测试，200K 次迭代，QEMU 大约 0.4 到 0.5 秒，tcg-rs 大约 0.32 秒，观察到快 20% 到 30%。

性能优势主要来自三个方面：执行循环优化贡献 8 到 10 个百分点，Guest 内存访问优化贡献 8 到 10 个百分点，数据结构和其他优化贡献剩余部分。

但必须诚实地说：我们目前只实现了原版大约 30% 的功能。功能少，代码路径短，快一点是符合预期的。这个数据的价值在于——**它验证了优化方向是正确的**。

---

## 第 12 页：差分测试——不信 AI，信参考实现 [1:00]

差分测试是最关键的一层质量保障。

工作流程很简单：tcg-rs 执行一条 RISC-V 指令，记录寄存器结果。QEMU 执行同样的指令，也记录结果。逐项对比。

核心原则是：我不信任 AI 的实现，我信任 QEMU 的实现。QEMU 经过 20 年打磨，可信度远高于 5 天写出来的代码。

项目总共 816 个测试，其中差分测试 35 个。有了差分测试这个安全网，就不需要逐行审查 AI 生成的代码。

> 【过渡】最后来谈一个更深层的问题——AI 的边界在哪里。

---

## 第 13 页：AI 的边界与人类核心价值 [1:00]

社区里有个朋友说了一句很形象的话：**初见即巅峰**。

什么意思？AI 生成的代码，第一印象很好——风格高、结构清晰，看起来像一个中级甚至接近高级工程师写的。但你去细究的时候，就会发现内部的具体问题了。就和相亲一样，我们看到的第一面就是最好的一面。

所以人类的核心价值在于两点：**定义问题的能力和验证结果的能力**。你的知识水平至少要和 AI 在同一水平，才能有效协作。否则你会被 AI 的初见印象「欺骗」。

商业级质量仍需人深度参与。AI 是非常好用的工具，但它代替不了人。

---

## 第 14 页：专家经验 × Agent 智力 [1:15]

前面讲了很多 Agent 的能力和边界，这里我想深入讨论一个问题：**专家经验在 Agent 协作中到底是正向的还是限制性的？**

答案是两面的，但以正向为主。

正向作用体现在方向判断上。tcg-rs 最典型的案例就是"后端先行"——我没有让 Agent 按教科书顺序从 IR 设计开始，而是基于之前团队研究 Houdini 的经验，直接先做 x86 后端。Agent 自己不具备这种路径判断能力。寄存器映射故事也是一样，"用 XMM 寄存器"这个非常规思路，Agent 自己不会想到。

但如果专家过度干预实现细节，就会把 Agent 当打字员用，浪费了它的 Plan Mode 和自纠错能力。所以关键是：**专家管"做什么"和"往哪走"，不管"怎么写"。**

那模型智力和专家经验是什么关系？我的观察是：**模型越强，对专家经验的最低要求反而越高。** 这很反直觉。因为模型越强，代码表面质量越好，越难发现深层问题。如果你水平不够，连它骗了你都不知道。用一个类比：模型智力是杠杆的长度，专家经验是支点。杠杆越长，支点覆盖的面积可以更小——你不需要管细节了——但支点的位置越关键。

最后看这张表，人的分工是动态的：早期深度参与定路径，中期审设计文档，后期只管方向和验收。不是一成不变的，而是随信任度和任务复杂度动态调整。

---

## 第 15 页：三个关键实践 [1:00]

最后总结三个可复用的实践。

第一是 CLAUDE.md 驱动。把它当成项目宪法，三四轮对话就主动重启，靠文档恢复上下文。成本更低，准确性更高。

第二是差分测试加 Agent 自我迭代。让 Agent 自己写测试、发现 bug、自己修。50% 的代码是测试——这是 AI 生成代码的必要投入。

第三是多 Agent 交叉验证。不需要复杂框架，两个终端、人在中间调度、不同模型互相卷。

这三个实践不是 tcg-rs 专属的，任何系统级的 AI 辅助开发项目都可以借鉴。

> 【过渡】那能不能把这三个实践系统化，变成一个可复用的工具？

---

## 第 16 页：从手动到自动——Humanize 插件 [1:15]

做完 tcg-rs 之后，我一直在想一个问题：我手动做的这些事——双终端调度、设计文档驱动、Agent 自测自修——能不能系统化？

Humanize 就是这个思路的产物。它是一个 Claude Code 插件，把我们前面讲的三个实践自动化了。

第一，它把人工调度双终端变成了 RLCR 循环——Claude 实现代码，Codex 做独立审查，发现问题就反馈回来继续迭代，直到 Codex 认可输出 COMPLETE。人不需要手动在两个终端之间切换了。

第二，它把设计文档驱动变成了结构化的计划生成。你写一份草稿，gen-plan 会把它转化成带 AC-X 验收准则的计划，每个准则都有 TDD 风格的正测试和负测试。这比我之前手写的 CLAUDE.md 更严谨。

第三，也是最重要的——它加了防漂移机制。有一个 Goal Tracker 追踪所有任务状态，每 5 轮做一次完整对齐审查，防止 Agent 在长时间迭代中偏离目标。还记得前面说的"初见即巅峰"吗？Humanize 的多层审查就是为了解决这个问题。

核心理念三句话：**迭代优于完美，独立审查防盲点，多层防护防漂移**。

## 第 17 页：项目数据总览 [0:30]


最后放一下项目的关键数据：5 天，约 4.5 万行 Rust（含生成代码和测试），816 个测试，10 个 Crate，184 条 RISC-V 指令翻译，实测比 QEMU 快 20% 到 30%，花了 200 美元。

核心观点是：AI 已经具备系统级软件工程能力了。关键不在于 AI 写代码有多强，而在于我们人类定义问题和验证结果的能力有多强。
